DataHarbour Project (DHP) Comprehensive Blackbook

In-depth technical, architectural, and business documentation.

1. Executive Summary

DataHarbour Project (DHP) is an enterprise-grade, API-first data platform specifically engineered to run, orchestrate, and manage Apache Spark jobs with stringent operational controls. It is designed to bridge the gap between complex big data engineering and modern microservices-based software delivery.

By providing out-of-the-box job submission, a strong metadata catalog for database and table schema evolution, integrated object storage management, and comprehensive observability pipelines, the platform delivers a reliable and scalable foundation for data teams.

Its architecture employs a decoupled, event-driven pattern using Apache Kafka and Kubernetes, ensuring that the control plane remains responsive and highly available even during intensive data processing workloads in the execution layer.

2. Business Details & Motivation

The Problem Statement

Traditional data engineering teams often struggle with operationalizing Apache Spark workloads. Issues such as unmanaged job submissions, lack of dependency tracking, opaque failure states, and fragmented logging make data operations difficult and costly. Moreover, data governance and schema evolution are frequently treated as afterthoughts, leading to data swamps.

The Solution

DataHarbour Project (DHP) solves this by bringing standard software engineering practices to big data. It abstracts the complexity of Kubernetes and Spark configurations behind RESTful APIs. Data scientists and engineers can submit jobs via a simple HTTP request without needing to understand the underlying infrastructure.

Key Business Value Propositions

  • Developer Velocity: API-first design allows seamless integration into existing CI/CD pipelines and custom internal portals.
  • Operational Resilience: Built-in retry mechanisms, dead-letter patterns, and terminal state tracking prevent jobs from failing silently.
  • Cost Efficiency: Isolated Spark runtimes on Kubernetes scale up on demand and terminate upon completion, minimizing idle compute costs.
  • Data Governance: The centralized Metadata Service enforces schema controls and versioning, critical for regulatory compliance and data quality.
  • Extensibility: Microservices architecture allows independent scaling of job orchestration, metadata management, and storage.

Design Goals

  1. API-first job operations: All interactions occur via documented REST APIs.
  2. Isolated runtime per job: Every workload runs in a pristine environment.
  3. Durable asynchronous orchestration: Kafka ensures no jobs are lost during API restarts.
  4. Clear metadata governance: Dedicated service for schema and database management.
  5. Production-style observability: Out-of-the-box integration with Prometheus and Grafana.

5. Control Plane & Orchestration Deep Dive

Job Submission Lifecycle

The orchestration workflow is robust against failures. When a client submits a job:

  1. The Job Service validates the request and writes a PENDING record to PostgreSQL.
  2. It reliably publishes an event to the Kafka topic spark-job-submissions. Upon success, the job is marked QUEUED.
  3. The Orchestrator consumes this message. It updates the state to PROVISIONING via an internal API.
  4. A Kubernetes Job is generated dynamically, injecting required S3 credentials, Spark configurations, and a unique X-Internal-Token.
  5. When the Spark container boots, it immediately makes a callback to the Job Service declaring itself RUNNING.
  6. Upon completion, the Spark entrypoint script catches the exit code and makes a final callback reporting SUCCESS or FAILED.

Retry and Fault Tolerance Model

Rather than relying purely on Kubernetes' internal backoff mechanisms (which lack business context), the retry logic is handled at the application layer.

If a job fails and retry_count < max_retries, the Job Service automatically increments the counter and requeues the job on Kafka. If the retries are exhausted, the job is explicitly marked DEAD.

6. Data Plane & Metadata Management

Metadata Catalog Architecture

The Metadata Service acts as the brain for data governance. It abstracts the underlying storage mechanisms (like Iceberg or Delta) by persisting logical definitions in PostgreSQL.

  • Namespaces: Logical grouping of tables (catalog_databases).
  • Tables: Defines the table formats (catalog_tables), e.g., ICEBERG.
  • Schema Evolution: Stored in schema_history. Every change (add column, drop column, change type) creates an immutable history record.

Storage Interoperability

In local development, MinIO is used as an S3-compatible backend. A bootstrap script provisions essential buckets:

  • lakehouse-warehouse: Stores optimized analytical datasets.
  • lakehouse-raw: Landing zone for incoming messy data.
  • lakehouse-scripts: Artifact repository for PySpark entrypoint scripts.
  • lakehouse-logs: Optional cold storage for logs.

Spark jobs interact with this layer exclusively via the s3a:// protocol, ensuring zero code changes when migrating from MinIO to AWS S3.

7. Observability & Monitoring

A production system requires deep visibility. DataHarbour Project (DHP) embeds observability directly into its DNA.

Metrics Pipeline

All FastAPI services expose a /metrics endpoint instrumented via prometheus_fastapi_instrumentator. Prometheus scrapes these endpoints alongside blackbox probes (HTTP/TCP health checks).

Key metrics collected:

  • API Request Rate and P95 Latency
  • Service Resource Consumption (CPU, Memory, File Descriptors)
  • Kafka Consumer Lag
  • Job State Transitions

Logging Pipeline

Logs are critical for debugging distributed Spark tasks. When a Kubernetes Spark pod runs:

  1. Spark writes its stdout/stderr to /var/log/spark/<job_id>.log.
  2. A Fluent Bit sidecar container (or DaemonSet) tails these files.
  3. Fluent Bit enriches the logs with labels (job_id, container_id) and forwards them to Loki.
  4. Users fetch logs cleanly via the Log Service API (/api/v1/logs/{job_id}).

Grafana Dashboards

The platform provisions an extensive "Lakehouse Admin Ops" dashboard out-of-the-box (UID: lakehouse-admin-ops). It aggregates API throughput, latency histograms, error rates, and a live stream of critical errors from Loki.

8. Security Model

Security is implemented at both the edge and the internal boundaries.

Authentication Mechanisms

  • External Access: All business endpoints require an X-API-Key. This protects job submissions, data definitions, and log reads.
  • Internal Access: Microservice-to-microservice communication (like the Spark callback to Job Service) requires an X-Internal-Token. This prevents external actors from spoofing job statuses.
  • Timing Attack Prevention: Token validation relies on hmac.compare_digest for constant-time comparison.

Future Hardening

The architectural roadmap includes migrating to OIDC-backed Identity Providers, integrating external Secret Managers (like HashiCorp Vault), and enforcing strict mTLS between services using a service mesh (e.g., Istio).

Common Troubleshooting

Scenario: Jobs stuck in PENDING or QUEUED

  • Check Kafka health: docker logs lakehouse-kafka --tail 100
  • Check Orchestrator: docker logs lakehouse-orchestrator --tail 200
  • Verify Kafka topic auto-creation is enabled in the cluster.

Scenario: Orchestrator Cannot Reach Kubernetes

  • Ensure ~/.kube/config is correctly mounted.
  • Ensure K8S_HOST_ALIAS=host.docker.internal and K8S_SKIP_TLS_VERIFY=true are configured for local Docker-to-host connectivity.

Scenario: Missing Job Logs

  • Verify Loki is operational: curl -s http://localhost:3100/ready
  • Ensure the Fluent Bit sidecar is successfully parsing and shipping logs from the Spark pod.

10. API Reference Highlights

Full OpenAPI specifications are available at /docs for each service port. Below are critical endpoints.

Job Service (Port 8001)

  • POST /api/v1/jobs/: Submit a new job. Requires JSON payload with entrypoint, Spark config, and job_name.
  • GET /api/v1/jobs/{job_id}: Poll for status.
  • PUT /api/v1/jobs/{job_id}/status: (Internal) Update job status. Requires Internal-Token.

Metadata Service (Port 8002)

  • POST /api/v1/databases/{db_name}/tables/: Create a new logical table definition. Supports nested JSON schema declarations.
  • GET /api/v1/databases/{db_name}/tables/{table_name}/snapshots: Retrieve history of schema modifications.

Storage Service (Port 8004)

  • POST /api/v1/storage/buckets/{bucket_name}/presigned-url: Generate secure, time-bound URLs for uploading large Spark scripts or data files without exposing S3 credentials.
